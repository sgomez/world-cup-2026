# Source-agnostic real Live Feed

The **Live Feed** port (ADR 0019) was realized only by a mock. We now add a real Feed that obtains each `live` match's current scoreline and finished state from an external page addressed by the match's **Match Link**, selected at bootstrap by the `LIVE_FEED_SOURCE` environment variable and defaulting (fail-safe) to the mock. Adding the adapter behind the existing port is routine; the decision worth recording is that the data **source itself is kept out of the repository**.

The platform names no specific provider. The selection value is neutral (`remote`), the adapter is named for its mechanism (`RemoteLiveFeed`) not its origin, and no committed code or document identifies where the data comes from. The only source-revealing artifacts — the link-discovery and link-seeding scripts and the generated links file — live outside version control (gitignored) and are run operationally to populate `LiveResult.link` in the production database. The running app reads the source purely through those stored links; the link is per-match opaque data the Feed dereferences.

We accept this because the source is unstable and replaceable — two prior real-feed attempts against named third parties were abandoned when those sources proved unreliable. Binding the repository's vocabulary, code, or documentation to any one provider invites the same dead-end, and committing the source URL would make a future swap a code-and-docs migration rather than an environment change plus a re-seed. Keeping the source agnostic lets a future Feed be a new sibling adapter and an env value, with the port, the Poller, the upsert command, and the domain untouched.

The trade-off is that the repository cannot self-document where live data originates: that operational knowledge lives in the gitignored scripts and the production environment, not in the code. We judge a clean, swappable boundary worth more than in-repo self-documentation of a source we expect to change.
